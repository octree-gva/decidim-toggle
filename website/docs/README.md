# Documentation structure

Structure map for `website/docs/` (`routeBasePath: /`). Same rules site-wide — no one-off page layouts.

## Integrator reading order

Module developers adding a settings tab:

1. [Overview](./index.md) — what toggle does, host install
2. [Integrate](./integrate/index.md) — Form + Command contract
3. [Add a settings tab](./integrate/quickstart.md) — form, engine registration, verify
4. [Attributes](./integrate/attributes.md) — field types, helptext, callouts (when the default tab is not enough)
5. [Troubleshooting](./integrate/troubleshooting.md) — when something breaks
6. [API surface](./reference/api-surface.md) — lookup while coding (any step)

Contributors hacking **decidim-toggle**: [Contribute](./developer/contribute.md) → [Documentation site](./developer/documentation.md).

## Page shape

1. **Overview** — what, who reads it
2. **Context** — how it fits (optional one diagram or table)
3. **Detail** — steps, tables, troubleshooting
4. **See also** — links to logical next/back pages

Default integrator path is always Form + Command. No decision trees.

## Roles (doc maintenance)

| Role | Owns |
|------|------|
| **Positioning specialist** | README opening, market category, anti-persona |
| **Wiki** | This map, C4 order, cross-links, `yarn build` |
| **Bored developer** | Step count ≤5 per guide; cut fluff before merge |
| **Developer experience specialist** | Docker / `./bin/check` truth in README and contribute |
| **Maintainer** | API drift vs code, migration notes, release scope |

Flow: positioning → wiki restructure → bored developer gate → DX truth pass.
